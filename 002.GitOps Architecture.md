# Recap of GitOps

GitOps uses:

* Git as the single source of truth
* Kubernetes manifests stored declaratively
* Controllers to synchronize Git state with Kubernetes cluster state

GitOps continuously ensures:

* Kubernetes state = Git state

# Core GitOps Flow

## Step 1

Developer pushes YAML manifests to Git.

## Step 2

GitOps controller reads manifests.

## Step 3

Controller deploys resources to Kubernetes.

## Step 4

Controller continuously compares:

* Git state
  vs
* Kubernetes cluster state

## Step 5

If drift is detected:

* Controller reconciles the cluster automatically.

---

# Reconciliation Logic

Reconciliation is the most important concept in Kubernetes controllers.

## Definition

Reconciliation means continuously ensuring:

> Desired state = Actual state

In GitOps:

* Desired state → Git repository
* Actual state → Kubernetes cluster

The GitOps controller constantly compares both states.

---

# Example of Reconciliation

Suppose:

* Someone manually modifies a deployment inside Kubernetes.

GitOps controller behavior:

1. Detects the difference
2. Reads correct configuration from Git
3. Restores Kubernetes state

This process is called:

* Drift correction
* Auto healing
* Continuous reconciliation

# Argo CD Architecture

Argo CD consists of multiple microservices.

The architecture is designed to:

* Read Git state
* Read Kubernetes state
* Compare both states
* Synchronize differences

---

# Main Components of Argo CD

## 1. Repo Server

### Responsibility

Connects to Git repositories.

### Functions

* Pulls manifests from Git
* Reads desired configuration
* Stores Git state information

### Purpose

Acts as the Git state provider.

---

## 2. Application Controller

### Responsibility

Connects to Kubernetes cluster.

### Functions

* Reads Kubernetes cluster state
* Compares Git state with cluster state
* Performs reconciliation
* Applies manifests to Kubernetes

### Core Logic

This component performs:

* Sync operations
* Drift detection
* Auto healing

---

# Application Controller Workflow

## Step 1

Gets desired state from Repo Server.

## Step 2

Gets actual state from Kubernetes.

## Step 3

Compares both states.

## Step 4

If mismatch exists:

* Synchronizes cluster with Git.

---

## 3. API Server

### Responsibility

Provides user interaction layer.

### Supports

* Web UI
* CLI access
* API communication

### Functions

* Authentication
* Authorization
* User requests
* Cluster management

Users communicate with Argo CD through the API Server.

---

## 4. Dex

### Purpose

Provides authentication integration.

### Supports

* OAuth
* OIDC
* SSO integrations

### Examples

Can integrate with:

* Google login
* LDAP
* GitHub
* Other identity providers

Dex acts as an authentication proxy.

---

## 5. Redis

### Purpose

Caching layer.

### Why Needed?

Argo CD components maintain cluster state information.

Redis helps:

* Store cached state
* Improve performance
* Preserve state during failures

---

# Stateful Nature of GitOps Controllers

GitOps controllers maintain:

* Cluster state
* Git state
* Synchronization metadata

Therefore:

* Components are often stateful
* Persistent caching is required

Redis helps maintain this information.
