# 4. Install Argo CD Using Plain YAML

## Step 1 — Create Namespace

```bash
kubectl create namespace argocd
```

Why separate namespace?

Because controllers should be isolated:

* Easier troubleshooting
* Better organization
* Cleaner management

Examples:

* Istio → `istio-system`
* Argo CD → `argocd`

---

## Step 2 — Install Argo CD Components

```bash
kubectl apply -n argocd -f https://raw.githubusercontent.com/argoproj/argo-cd/stable/manifests/install.yaml
```

This installs:

* CRDs
* Deployments
* Services
* RBAC
* ConfigMaps
* Secrets
* Controllers
* Redis
* API Server

# 6. Important Argo CD Components

---

## A. argocd-server

Purpose:

* Main API server
* Handles:

  * UI requests
  * CLI requests

Think of it like:

* Kubernetes API Server

---

## B. repo-server

Purpose:

* Connects to Git repositories
* Reads manifests

Supports:

* GitHub
* GitLab
* Bitbucket
* Any Git-based VCS

---

## C. application-controller

MOST IMPORTANT COMPONENT.

Purpose:

* Core reconciliation engine

Responsibilities:

* Compare:

  * Git state
  * Kubernetes cluster state
* Detect drift
* Sync cluster automatically

Core GitOps principle:

> Kubernetes cluster must always match Git repository.

---

## D. Redis

Purpose:

* Caching
* State management

Needed because:

* Controllers are stateful
* Information must survive restarts

---

## E. Dex Server

Purpose:

* Authentication
* SSO integration

Supports:

* OIDC
* OAuth
* LDAP

Examples:

* Google login
* GitHub login
* Corporate SSO

---

## F. Notifications Controller

Introduced in newer versions.

Purpose:

* Send notifications
* Slack/email integrations

---

## G. ApplicationSet Controller

Advanced feature.

Purpose:

* Generate multiple Argo CD applications automatically

Useful when:

* Managing many clusters
* Multi-environment deployments

# 7. Accessing Argo CD UI

---

## Step 1 — Expose Service

Edit service:

```bash
kubectl edit svc argocd-server -n argocd
```

Change:

```yaml
type: ClusterIP
```

to:

```yaml
type: NodePort
```

---

## Step 2 — Create Tunnel (Minikube)

```bash
minikube service argocd-server -n argocd
```

This gives:

* HTTP URL
* HTTPS URL

---

## Step 3 — Open in Browser

Example:

```text
https://127.0.0.1:XXXXX
```

Ignore insecure certificate warning.


## Decode Password

```bash
kubectl edit secret argocd-initial-admin-secret -n argocd
```

Copy encoded password.

Decode:

```bash
echo <password> | base64 --decode
```

Use decoded password to login.

# 24. Key Interview Concepts

---

## Q1. What is GitOps?

GitOps is a deployment methodology where:

* Git is the single source of truth
* Kubernetes state continuously matches Git state

---

## Q2. What is Reconciliation?

Reconciliation means:

> Continuously comparing desired state and actual state.

Desired state:

* Git repository

Actual state:

* Kubernetes cluster

---

## Q3. What does Application Controller do?

* Core reconciliation engine
* Syncs Git and Kubernetes

---

## Q4. Difference Between Traditional CI/CD and GitOps?

Traditional CI/CD:

* One-time deployment

GitOps:

* Continuous reconciliation
* Auto-healing
* Drift detection

---

## Q5. Why Redis in Argo CD?

* Cache state
* Improve recovery
* Preserve controller information

---

## Q6. What is Dex?

Authentication component supporting:

* OIDC
* OAuth
* SSO integrations

---

## Q7. Why Use Automatic Sync?

Automatically applies Git changes to cluster.

# 25. MOST IMPORTANT MEMORY FLOW

Remember this complete flow:

```text
Developer pushes code/manifests to Git
        ↓
Repo Server reads Git
        ↓
Application Controller compares state
        ↓
If drift detected → Sync cluster
        ↓
Argo CD keeps cluster aligned with Git
```
